# ☀️ Solar Panel Historical Data Pipeline

A **data engineering project** that collects historical solar irradiance data from the **NASA POWER API**, stores it in **Google Cloud Storage (GCS)** and **BigQuery**, transforms it through a **Bruin** pipeline, and visualises the results in **Looker Studio**.

The project estimates the daily kWh production of a standard solar panel for one or more geographic locations over a configurable date range.

---

## 📐 Architecture Overview

```
NASA POWER API
      │
      ▼
[Python Ingest - Bruin]
      │
      ├──► GCS Bucket  (raw CSV files)
      │
      └──► BigQuery: solar_raw.nasa_irradiance
                │
                ▼
        [SQL Transform - Bruin]
        BigQuery: solar_staging.stg_solar_irradiance
                │
                ▼
        [SQL Transform - Bruin]
        BigQuery: solar_mart.solar_production
                │
                ▼
        Looker Studio Dashboard
```

### BigQuery Layers

| Layer | Dataset | Table | Description |
|---|---|---|---|
| Raw | `solar_raw` | `nasa_irradiance` | Raw NASA daily data per location |
| Staging | `solar_staging` | `stg_solar_irradiance` | Cleaned and typed fields, quality checks |
| Mart | `solar_mart` | `solar_production` | Estimated daily kWh per panel, cloud factor, temperature correction |

## ⚙️ Prerequisites

- **Docker Desktop** installed and running
- A **Google Cloud Platform** project with billing enabled
- The following **GCP APIs must be enabled** in your project:
  - `Identity and Access Management (IAM) API`
  - `Cloud Resource Manager API`
- A GCP **service account key** (JSON) with at least `Owner` or the following roles:
  - `BigQuery Admin`
  - `Security Admin`
  - `Service Account Admin`
  - `Service Account Key Admin`
  - `Storage Admin`
- **NASA POWER API** — no key needed, it is public

## 🚀 Setup Guide

### Step 1 — Clone the repository

```bash
git clone https://github.com/cookieman-dab/data-engeneering-project.git
cd data-engeneering-project
```

---

### Step 2 — Create the `secrets/` folder

Place your main GCP service account JSON key here. Terraform will use it to provision resources and will automatically generate a second key for Bruin.

```bash
mkdir secrets
# Copy your GCP service account key into secrets/
cp /path/to/your-sa-key.json secrets/sa.json
```

---

### Step 3 — Configure Terraform variables

Copy the example file and fill in your values:

```bash
cp terraform/terraform.tfvars.example terraform/terraform.tfvars
```

Edit `terraform/terraform.tfvars`:

```hcl
project_id         = "your-gcp-project-id"
bucket_name        = "solar-raw-noble-sun"
service_account_id = "bruin-sa"
```

> **Note**: `bucket_name` must be globally unique across all of GCS.

---

### Step 4 — Configure the `.env` file

Create a `.env` file in the project root:

```dotenv
# GCP
GCP_PROJECT_ID=your-gcp-project-id
GCP_BUCKET=solar-raw-noble-sun

# Pipeline parameters
START_DATE=2020-01-01
END_DATE=2024-12-31

# Solar panel specs
PANEL_AREA=1.7
PANEL_EFFICIENCY=0.20

# Locations to fetch (JSON array)
LOCATIONS=[{"id":"Roma","lat":41.9,"lon":12.5},{"id":"Milano","lat":45.5,"lon":9.2}]
```

| Variable | Description | Example |
|---|---|---|
| `GCP_PROJECT_ID` | Your GCP project ID | `my-solar-project` |
| `GCP_BUCKET` | GCS bucket name (same as in tfvars) | `solar-raw-noble-sun` |
| `START_DATE` | First date to fetch data for | `2020-01-01` |
| `END_DATE` | Last date to fetch data for | `2024-12-31` |
| `PANEL_AREA` | Panel surface area in m² | `1.7` |
| `PANEL_EFFICIENCY` | Panel efficiency (0–1) | `0.20` |
| `LOCATIONS` | JSON array of location objects | `[{"id":"Roma","lat":41.9,"lon":12.5}]` |

---

### Step 5 — Build the Docker image

```bash
docker compose build
```

This builds a single image containing:
- Python 3.11 with all `requirements.txt` dependencies
- Terraform 1.7.0
- Bruin CLI (latest)

---

### Step 6 — Provision GCP Infrastructure with Terraform

```bash
docker compose run --rm terraform
```

This runs `terraform init → plan → apply` automatically inside the container. It will create:

- ✅ A GCS bucket for raw CSV files
- ✅ Three BigQuery datasets: `solar_raw`, `solar_staging`, `solar_mart`
- ✅ A service account (`bruin-sa`) with the required IAM roles
- ✅ A service account key saved to `secrets/bruin-sa.json`

---

### Step 7 — Run the Bruin Pipeline

```bash
docker compose run --rm pipeline
```

The pipeline runs the following assets in order:

1. **`ingest.nasa_power_ingest`** (Python)
   - Calls the NASA POWER API for each location in `LOCATIONS`
   - Uploads raw CSV to GCS: `gs://<bucket>/raw/nasa_<location>_<start>_<end>.csv`
   - Loads parsed rows into `solar_raw.nasa_irradiance`

2. **`solar_staging.stg_solar_irradiance`** (BigQuery SQL)
   - Cleans and casts raw fields
   - Applies `not_null` and `positive` data quality checks
   - Outputs to `solar_staging.stg_solar_irradiance`

3. **`solar_mart.solar_production`** (BigQuery SQL)
   - Applies temperature derating: `1 + (−0.004 × (T − 25))`
   - Computes cloud factor: `irradiance / clear_sky_irradiance`
   - Estimates daily kWh: `irradiance × panel_area × efficiency × temp_factor`
   - Outputs to `solar_mart.solar_production`

> The pipeline is also scheduled to run automatically on the **1st of every month** (cron: `0 0 1 * *`).

## 📊 Looker Studio Visualisation

Connect Looker Studio to BigQuery using the `solar_mart.solar_production` table.

### Connecting to BigQuery

1. Go to [lookerstudio.google.com](https://lookerstudio.google.com)
2. Click **Create → Report**
3. Choose **BigQuery** as data source
4. Select **Custom query** (Query personalizzata)
5. Select your project
6. Paste your query (see examples below)
7. Click **Connect**

---

### Query 1 — Total kWh produced by location

Use this as a **Bar Chart** or **Scorecard** to compare locations:

```sql
SELECT
  location_id,
  COUNT(*) AS total_giorni,
  ROUND(SUM(estimated_kwh), 1) AS total_kwh_prodotto
FROM `solar_mart.solar_production`
GROUP BY location_id
ORDER BY total_kwh_prodotto DESC
```

| Column | Description |
|---|---|
| `location_id` | City/location name |
| `total_giorni` | Number of days with data |
| `total_kwh_prodotto` | Total estimated kWh produced over the period |

**Use as**: Bar chart with `location_id` on the X-axis and `total_kwh_prodotto` as the metric.

---

### Query 2 — Average daily kWh by month and location

Use this as a **Time Series** or **Heatmap** to see seasonal trends:

```sql
SELECT
  location_id,
  year_month,
  ROUND(AVG(estimated_kwh), 2) AS media_kwh_giorno_mese
FROM `solar_mart.solar_production`
WHERE year_month IS NOT NULL
GROUP BY location_id, year_month
ORDER BY location_id, year_month
```

| Column | Description |
|---|---|
| `location_id` | City/location name |
| `year_month` | Month in `YYYY-MM` format |
| `media_kwh_giorno_mese` | Average daily kWh for that location and month |

**Use as**: Time series chart with `year_month` on the X-axis, `media_kwh_giorno_mese` as the metric, and `location_id` as the dimension/breakdown.

## 🗑️ Destroying Infrastructure (Terraform Destroy)

Once you are done with the project and want to **delete all GCP resources** created by Terraform (bucket, BigQuery datasets, service account, IAM bindings), run:

```bash
docker compose run --rm terraform sh -c "
  terraform init -input=false &&
  terraform destroy -auto-approve -input=false
"
```

> [!CAUTION]
> This will **permanently delete** the following GCP resources:
> - The GCS bucket and **all raw CSV files** inside it
> - All three BigQuery datasets (`solar_raw`, `solar_staging`, `solar_mart`) and their tables
> - The `bruin-sa` service account and its IAM roles
> - The service account key (`secrets/bruin-sa.json`)

What is **not** deleted:
- Your local `secrets/sa.json` (your admin key — you manage this yourself)
- Your local `terraform/terraform.tfvars` and `.env` files
- The Docker image and containers

### Full teardown (infrastructure + local Docker resources)

If you also want to remove the Docker image and all stopped containers:

```bash
# 1. Destroy GCP infrastructure
docker compose run --rm terraform sh -c "terraform init -input=false && terraform destroy -auto-approve -input=false"

# 2. Remove Docker containers, images and volumes
docker compose down --rmi all --volumes
```

## 🛠️ Troubleshooting

If you run into issues during the setup, here are some common solutions:

- **Error 409: The requested bucket name is not available**  
  GCS bucket names must be globally unique across all Google Cloud users. If you get a `409 Conflict` error during `terraform apply`, change the `bucket_name` in `terraform/terraform.tfvars` and `GCP_BUCKET` in `.env` to something more unique (e.g., append your project ID or random numbers).

- **Error 403: Identity and Access Management (IAM) API has not been used...** or **Cloud Resource Manager API has not been used...**  
  Google Cloud blocks Terraform from creating resources if the necessary APIs are disabled. Click the link provided in the error message to enable the API, wait 5-10 minutes for it to propagate, and run `docker compose run --rm terraform` again.

- **Bruin pipeline fails with `no such file or directory` for `bruin-sa.json`**  
  This happens if the previous Terraform step failed. Make sure Terraform finishes successfully so it can generate the `secrets/bruin-sa.json` key needed by the pipeline.
